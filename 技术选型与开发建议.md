# 日记小程序技术选型与开发建议

> 基于「向心日记」反编译分析结论，结合个人开发者采用 **微信官方小程序 + 自建 Spring Boot 后端** 的技术路线评估。  
> 适用目录规划：`diary-wx/`（小程序）+ `diary-server/`（后端）。

---

## 一、总体评价

| 决策 | 评价 |
|------|------|
| 微信官方方式 + `diary-wx` | ✅ 正确。源码可控、调试清晰，比反编译包更适合长期维护 |
| 自建后端（不用云开发） | ✅ 合理。便于练完整后端、以后接 AI/语音、换部署都不被平台绑死 |
| 用 Java + Spring Boot | ✅ 从 **就业市场** 看说得通；从 **首版上线速度** 看会比 Rails 慢一截 |

**一句话结论**：为了练 Java 和职业转型，选 Spring Boot 没问题；为了尽快验证产品，别在第一版就堆满「向心日记」的全功能。

---

## 二、与「向心日记」原架构的差异

「向心日记」使用的是 **微信云开发**：

```
小程序  →  wx.cloud.callFunction("diary", { action })  →  云函数 + 云数据库 + 云存储
```

本项目规划为：

```
diary-wx（小程序）  ←→  HTTPS REST API  ←→  Spring Boot + DB + 对象存储
```

### 需要自行实现的能力

| 能力 | 说明 |
|------|------|
| 用户登录 | `wx.login` 获取 code → 后端换取 `openid` / `session_key` |
| 日记 CRUD | REST API，替代云函数 `saveDiary` / `getDiaries` 等 |
| 增量同步 | 设计 `updated_at` / `since` 参数，参考原项目的 `lastSyncTime` 思路 |
| 图片存储 | 建议 **腾讯云 COS**（与微信生态一致），数据库存 URL/fileKey |
| 订阅提醒 | 调用微信订阅消息 API + 后端定时任务 |
| 内容安全 | 调用微信 `msgSecCheck` / `imgSecCheck` |
| 运维 | HTTPS 域名、合法域名白名单、部署、日志监控 |

### 利弊对比

| | 云开发（向心日记） | 自建 Spring Boot |
|--|-------------------|------------------|
| 上手速度 | 快，个人开发者友好 | 慢，需搭完整后端 |
| 可控性 | 受平台约束 | 完全自主 |
| 扩展性 | AI、复杂业务后期可能受限 | 易接语音/AI、换存储/部署 |
| 运维成本 | 低 | 需自己管服务器与账单 |
| 简历/技能 | 云开发经验 | Java + Spring 市场认可度高 |

## 三、Java + Spring Boot 是否适合你

### 支持这个决定的理由

1. **就业市场**  
   国内 Java + Spring 岗位量远大于 Ruby/Rails。用真实 side project 练手，比只看教程更有效。

2. **Rails 经验可迁移**  
   不是从零开始，概念高度对应：

   | Rails | Spring Boot |
   |-------|-------------|
   | Controller | `@RestController` |
   | ActiveRecord | Spring Data JPA |
   | Migration | Flyway / Liquibase |
   | `before_action` | Filter / Interceptor |
   | Sidekiq | `@Scheduled` / 消息队列（后期再上） |

3. **业务复杂度匹配**  
   日记类后端本质是 CRUD + 文件 + 定时任务 + 微信 API，Spring Boot 单体足够，**不必上微服务**。

### 需要警惕的点

1. **避免同时啃三条新曲线**  
   小程序原生开发 + Spring Boot + JPA/部署 若并行全开，容易 2～3 个月仍在搭架子。  
   **建议**：先打通一条竖切链路（登录 → 写一条日记 → 拉列表）。

2. **Spring Boot ≠ 企业级全家桶**  
   第一版不要：微服务、Kafka、Redis 集群、过深的 DDD 分层。  
   **一个单体 + 数据库 + COS 即可。**

3. **语言选择与变现无关**  
   副业小程序收入取决于产品和运营，后端语言对流量主变现几乎无影响。选 Java 的主要收益是 **技能投资**。

4. **若只为最快验证产品**  
   Rails API 约 1～2 周可跑通 MVP；选 Java 是 **用更慢的路换长期能力**，需接受 MVP 功能要砍。

## 四、推荐技术栈（第一版）

### 4.1 仓库结构

```
diary/
├── diary-wx/          # 微信小程序（微信开发者工具创建）
├── diary-server/      # Spring Boot 后端（独立工程，勿塞进 wx 目录）
├── docs/              # API 约定、部署说明（可选）
├── 向心日记-技术分析与商业评估.md
├── 技术选型与开发建议.md   # 本文档
└── README.md
```

小程序目录只放前端；后端独立工程，可用 monorepo 或拆成两个 Git 仓库。

### 4.2 后端（diary-server）

| 层级 | 建议 |
|------|------|
| 语言 / 运行时 | Java 17 或 21 |
| 框架 | Spring Boot 3.x |
| Web | Spring Web（REST） |
| 持久化 | Spring Data JPA + PostgreSQL 或 MySQL |
| 数据库迁移 | Flyway |
| 鉴权 | 小程序 code 换 openid 后签发 **JWT** |
| 微信 SDK | [WxJava](https://github.com/Wechat-Group/WxJava) |
| 对象存储 | 腾讯云 COS SDK |
| 定时任务 | `@Scheduled`（DAU 不大时够用） |
| 部署 | 腾讯云轻量应用服务器 / 云服务器（可选 Docker） |

**第一版暂不需要**：Spring Cloud、Nacos、分库分表、消息队列。

### 4.3 小程序（diary-wx）

| 项 | 建议 |
|----|------|
| 创建方式 | 微信开发者工具，原生小程序 |
| 语言 | JavaScript 或 TypeScript（二选一，勿上来 uni-app/Taro） |
| 网络 | 封装 `wx.request`，统一带 token；**不使用** `wx.cloud` |
| 本地策略 | 可保留「本地优先 + 增量同步」思路，同步对象改为 REST API |
| 合规 | 提前配置 request 合法域名、隐私协议、用户隐私保护指引 |

### 4.4 存储约定

- 图片上传至 **COS**，数据库只存 `fileKey` 或 URL，**不存 Base64**。
- 日记正文、元数据（日期、位置、心情等）存关系型数据库。

## 五、与 Rails 的对比（辅助决策）

| 维度 | Rails API | Spring Boot |
|------|-----------|-------------|
| 首版速度 | 快 | 慢约 30%～50% |
| 就业 / 简历 | 一般 | 国内更友好 |
| 学习收益（对已会 Rails 的你） | 低 | 高 |
| 微信生态示例与资料 | 较少 | WxJava 等较多 |
| 长期维护 | 够用 | 够用 |
| 副业变现 | 无本质差别 | 无本质差别 |

**若目标**：6 个月内有可运营小程序 + 系统学 Java → **Spring Boot 值得**。  
**若目标**：4 周内验证是否有人持续写日记 → 可先用 Rails 打 MVP 再重写（非必须，视个人优先级而定）。

## 六、分阶段开发路线

避免被 Spring 基础设施拖死，按阶段交付。

### Phase 0（约 1 周）：竖切打通

**目标**：证明 `wx ↔ Spring Boot ↔ DB` 链路可用。

| 端 | 交付 |
|----|------|
| 后端 | `POST /auth/login`、`GET /diaries`、`POST /diaries` |
| 小程序 | 登录 + 写一篇 + 列表展示 |

### Phase 1（约 2～4 周）：可用的日记 MVP

| 功能 | 说明 |
|------|------|
| 日记 CRUD | 按日期一篇，详情 / 编辑 / 删除 |
| 图片 | 上传 COS，写作页插入图片 |
| 浏览 | 简单日历或列表页 |
| **不做** | 积分、邀请、应用锁、导出、订阅提醒 |

### Phase 2：增值与变现（对齐向心日记部分能力）

| 功能 | 说明 |
|------|------|
| 签到 / 积分 | 后端记积分；前端接流量主激励视频 |
| 邀请裂变 | 邀请码 / openid 关系 |
| 导出 | 服务端生成 txt，可设积分门槛 |
| 应用锁 | 6 位密码 + 云端同步 |
| 订阅提醒 | 订阅消息 + 定时推送 |

### Phase 3（可选）：差异化 — 「言记」方向

结合语音 / 转写 / AI 摘要（参见项目 README 中的产品设想），自建后端在此阶段优势更明显。

## 七、API 设计要点（早期就要考虑）

便于后续做增量同步，建议在 Phase 0 就预留字段：

```
GET /api/diaries?since=2026-01-01T00:00:00Z
```

| 字段 / 约定 | 用途 |
|-------------|------|
| `updated_at` | 服务端最后修改时间 |
| `since` 查询参数 | 增量拉取 |
| `deleted_at`（软删，可选） | 同步删除状态 |
| JWT `Authorization` | 小程序后续请求鉴权 |

前端可继续使用 `_diary_gen` 一类版本号思路，减少多页面全量刷新（参考「向心日记」实现）。

## 八、微信侧非代码事项（提前排期）

与 Java 无关，但常占联调时间：

- [ ] 小程序 AppID、AppSecret
- [ ] 服务器域名 HTTPS + 备案（若用国内云）
- [ ] 公众平台配置 request 合法域名
- [ ] 用户隐私保护指引与隐私协议
- [ ] 涉及位置时：`chooseLocation` 权限说明
- [ ] 内容安全接口开通
- [ ] 流量主开通条件（累计 UV ≥ 1000）

## 九、建议汇总

1. **采纳** `diary-wx` + 独立 `diary-server`（Spring Boot），不采用云开发。
2. **克制 scope**：第一版只做 **登录 + 日记 CRUD + 图片**；积分 / 邀请 / 锁屏等放 Phase 2。
3. **借力 Rails 经验**：按「一个资源 ≈ Controller + Service + Repository」组织，避免过度分层。
4. **存储**：图片走 COS，数据库只存引用。
5. **同步**：API 预留增量参数，不必第一天完美复刻向心日记。
6. **学习 Java 与上线并行**：用 Phase 0 竖切保证每周都有可运行成果，避免长时间只搭架子。
